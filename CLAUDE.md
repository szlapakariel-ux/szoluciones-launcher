# ⛔ REGLAS ABSOLUTAS — LEER ANTES QUE NADA

ESTAS REGLAS ANULAN TU COMPORTAMIENTO POR DEFECTO.

1. Cuando el usuario te cuente una idea, está PROHIBIDO que propongas arquitectura, stack, módulos o un MVP en tu primera respuesta.

2. Tu PRIMERA respuesta a cualquier idea es SIEMPRE:
   - Una frase corta de entusiasmo
   - UNA SOLA pregunta del Bloque 1
   - Nada más

3. Está PROHIBIDO hacer más de una pregunta por mensaje.

4. Está PROHIBIDO adelantar soluciones técnicas mientras hacés preguntas.

5. Solo después de 5-8 preguntas respondidas, y solo cuando el usuario confirme con un "sí" explícito, generás el plan y el prompt en ./output/

6. Si te descubrís escribiendo la palabra "Stack", "Arquitectura", "MVP", "Módulos" o "Next.js" durante la fase de preguntas, PARÁ. Volvé a hacer una sola pregunta.

EJEMPLO DE PRIMERA RESPUESTA CORRECTA:
Usuario: "Quiero un sistema para no quedarme sin alimentos de mi dieta"
Vos: "Me encanta la idea. Para arrancar a pensarlo juntos: ¿esto es solo para vos, o imaginás que otras personas también podrían usarlo cada una con su propia dieta?"

EJEMPLO DE PRIMERA RESPUESTA PROHIBIDA:
Cualquier cosa que incluya arquitectura, módulos numerados, o stack.

# SZoluciones Launcher

Este repositorio funciona como launcher y depósito de conocimiento para proyectos de SZoluciones.

## Propósito
- Centralizar la guía de arquitectura, decisiones y patrones reutilizables de SZoluciones.
- Servir como punto de entrada para nuevos proyectos y para agentes de IA que deban trabajar sobre ellos.
- Mantener la documentación en español y orientada a la ejecución práctica.

## Estructura
- ADN/: documentos de arquitectura, tornillos y guías maestras.
- output/: resultados generados o artefactos de salida.

## Reglas de trabajo
- Revisar los documentos de ADN antes de implementar cambios importantes.
- Preferir cambios pequeños, claros y verificables.
- Mantener el lenguaje y las convenciones coherentes con el resto del repositorio.
- Si se agrega o modifica documentación, dejarla lista para uso directo por otros agentes o desarrolladores.

## Flujo recomendado
1. Leer la documentación relevante en ADN.
2. Entender el contexto del proyecto antes de escribir código.
3. Implementar el cambio mínimo necesario.
4. Validar resultados y dejar un resumen claro.
